# REFLECTION

> 本文件由学生本人完成，AI 只参与润色和结构整理。最终观点和案例由本人确认。

## Superpowers 哪些技能最有用

这次项目里，我认为最有用的是 `test-driven-development` 和 `verification-before-completion`。TDD 让每个功能都有明确的验收条件，例如 Action parser、Guardrail、Gate、Runner 都是先写失败测试，确认缺失模块或行为不满足，再实现最小代码。这样 AI 不是“看起来写完了”就算完成，而是必须被测试验证。

`verification-before-completion` 也很重要。它要求每次说某个任务完成之前，都要重新运行对应测试和全量回归。这个流程虽然比较繁琐，但能减少 AI 生成代码后没有验证就继续推进的风险。对我来说，它相当于把“感觉没问题”改成了“有命令输出证明没问题”。

## TDD 在 AI 协作下的作用

我理解 TDD 是我控制 AI 输出的一种手段。AI 很容易给出看起来合理的实现，但这种合理可能只是表面上的。红灯阶段可以证明测试确实能抓到缺失功能，绿灯阶段可以证明当前实现满足了这一轮需求，最后的全量回归可以证明没有破坏之前已经完成的模块。

Task 5 的 Gate 实现给我印象比较深。第一次实现后测试失败，因为失败摘要里没有包含“至少 10 个”这个修复提示。这个问题如果人工只看代码可能不容易注意到，但测试直接暴露出反馈信息不适合后续修复闭环。后来我们不是随便改，而是先定位原因：摘要只取前几个 issue，导致节点数量问题被挤出去了。这个例子说明，TDD 不只是检查代码能不能跑，也能检查机制设计是否真的服务于后续流程。

## Subagent 工作流的有效边界

这次我没有大量使用真正并行的 subagent，但使用 Gemini 做了一次冷启动验证。做法是只给它 `SPEC.md` 和 `PLAN.md`，不提供之前的聊天记录，让它模拟一个没有上下文的新 agent 来判断计划是否能执行。这个过程帮助我检查文档本身是否足够清楚，而不是依赖当前对话里的隐含共识。

Gemini 主要指出了两个问题：第一，Task 3 中部分测试使用 `Path.cwd()`，隔离性不如统一使用临时目录；第二，凭据部分在 `SPEC.md` 里描述得比较完整，而 `PLAN.md` 的 MVP 实现只做 `credential_status()` 的 fail-closed 存根，二者的精细度需要对齐。我们采纳了这两个反馈，修改了测试隔离方式，也在规约里补充说明了 MVP 阶段凭据功能的边界。

这让我觉得 subagent 更适合做审查、冷启动验证和发现文档歧义，而不适合在需求还没有澄清时直接并行写代码。真正实现时，连续上下文、人工确认和 TDD 仍然更重要。

## SPEC / PLAN 质量如何影响实现

一开始我发现 `SPEC.md` 这个名字容易产生混淆。根目录的 `SPEC.md` 是课程项目的设计文档，描述的是 SpecGate 这个 harness 本身；但 harness 运行时的输入其实应该是一个 HTML 页面任务，例如 `TASK_SPEC.md` 加 `CHECKLIST.md`。如果不澄清这一点，agent 很容易把“项目规约”和“被生成页面的需求”混在一起。

后来我们明确了两类 SPEC 的边界：根目录 `SPEC.md` 是项目交付物，示例任务里的 `TASK_SPEC.md` 才是 harness 的运行输入。这个澄清让后续实现没有跑偏。`PLAN.md` 也起到了很大作用，它把任务拆成 Task 1 到 Task 10，并写清楚每个任务要创建哪些文件、先运行什么测试、预期失败是什么、通过后怎么提交。因此实现过程基本是按计划推进，而不是边写边猜。

## 最有效的 prompt / context 策略

我觉得最有效的策略是先把 MVP 边界说清楚。我明确要求项目只做 Python CLI harness、`MockLLM`、静态 HTML 生成/修复、Checklist/Gate 闭环和静态 Web 报告；同时明确不开放 shell、不做 Playwright、不做复杂前端。这样 AI 的实现空间被限制在课程要求和可控范围内。

另一个有效策略是要求 AI 在每次修改前说明：要改哪些文件、代码作用是什么、测试证据是什么。这样我可以参与开发过程，而不是只看到最后一大段代码。给 Gemini 做冷启动验证时，也只提供 `SPEC.md` 和 `PLAN.md`，用来检验文档是否自洽。这种 context 控制比单纯让 AI 自由发挥更可靠。

## 凭据与分发要求带来的工程思考

这个项目的 MVP 使用 mock 模式，因此不需要 API key。真实 LLM 如果后续加入，也不应该随便读取 `.env`，更不能把 key 打印到日志或 trace 里。所以我们在 MVP 中只实现了 `credential_status()` 的 fail-closed 行为：mock provider 可以安全运行，真实 provider 默认不配置、不允许运行。

Docker 和 CI 的要求让我意识到，课程项目不能只是“我本机能跑的脚本”。`Dockerfile`、`.gitlab-ci.yml` 和单元测试让项目更接近一个可交付的软件工程项目。Docker 验证一开始因为配置目录权限和网络连接失败，但后来在本机打开 Docker 并设置代理环境变量后，`docker pull`、`docker build` 和 `docker run` 都完成了验证。这个过程也被记录在 `AGENT_LOG.md` 里，而不是只保留成功结果。

## 如果重做会改变什么

如果重做一次，我会更早确定 GitHub Pages 或 GitLab Pages 的部署方式。因为项目最后需要一个静态 Web 报告 URL，而这个和仓库平台、分支、目录结构都有关系，越早确定越容易安排。

我也会更早加入 GitHub Actions。现在仓库实际托管在 GitHub，而 `.gitlab-ci.yml` 主要是满足课程文件要求。如果老师要求在平台上看到真实运行的 CI，GitHub Actions 会更直接。另外，`REFLECTION.md` 应该边做边写，不应该最后才集中补。Docker 也应该更早验证，因为权限、代理和镜像拉取问题都属于工程交付的一部分。

## 对 Superpowers 方法论的批判

Superpowers 的优点是规范、可追踪，并且能减少 AI 随机发挥。通过 SPEC、PLAN、TDD、验证和日志，每一步都有证据，最后也能解释项目是怎么一步步完成的。

但它也有明显成本。对于一个小型项目来说，这套流程会显得比较重：文件多、提交多、日志多，每个小功能都要走红绿流程。TDD 很稳，但确实增加了开发过程的长度。另外，如果前期计划写得不好，AI 可能会非常高效地执行一个错误计划。因此我认为 Superpowers 更适合目标明确、边界清楚的工程任务；如果需求还在探索阶段，应该先做充分的需求澄清，而不是急着进入执行流程。
