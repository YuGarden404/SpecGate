# REFLECTION

> 本文件由学生本人完成，AI 只参与润色和结构整理。最终观点和案例由本人确认。

## Superpowers 哪些技能最有用

这次项目中，我认为最有用的是 `test-driven-development` 和 `verification-before-completion`。Action parser、Guardrail、Gate、Runner 等功能都先用失败测试定义验收条件，再实现最小代码；声称完成前还要重新运行聚焦测试和全量回归。它们共同把“看起来没问题”变成“有命令输出证明没问题”。

## TDD 在 AI 协作下的作用

我把 TDD 看作控制 AI 输出的手段：红灯证明测试能捕捉缺失行为，绿灯证明本轮需求得到满足，全量回归则检查旧模块没有被破坏。Gate 初次实现时，失败摘要没有包含“至少 10 个”的修复提示，根因是摘要只截取前几个 issue，节点数量问题被挤出。这个案例说明，TDD 不只检查代码能否运行，也检查反馈是否真的服务于下一轮修复。

## Subagent 工作流的有效边界

项目早期我主要使用 Gemini 做冷启动验证：只提供 `SPEC.md` 和 `PLAN.md`，不提供聊天记录，让它检查计划能否被陌生 agent 独立执行。它指出测试目录隔离和凭据规格精细度不一致，帮助我发现文档依赖了对话中的隐含共识。

`Context Harness Deepening` 阶段后来使用独立的实现、规格和质量审查 agent；Gate/HITL 之后，因为任务会共同修改 Runner、Web 状态和过程材料，我选择在主线程按 `Inline Execution` 连续推进。这个变化让我认识到，subagent 是否有效取决于任务耦合度：边界清楚的审查适合委派，共享状态较多的任务则更需要连续上下文、人工确认和 TDD。

## SPEC / PLAN 与上下文策略

项目早期最容易混淆的是两类 SPEC：根目录 `SPEC.md` 描述 SpecGate harness，示例目录中的 `TASK_SPEC.md` 和 `CHECKLIST.md` 才描述一次 HTML 任务。明确这条边界后，agent 不会把项目规约当成运行输入。`PLAN.md` 又把工作拆成文件、失败测试、预期结果和验证命令，使实现能按可检查步骤推进，而不是边写边猜。

我觉得最有效的 context 策略是先明确 MVP：使用 Python CLI、`MockLLM`、静态 HTML 和 Checklist/Gate 闭环，不开放 shell，也不做 Playwright。后续增加 Web 产品壳时仍保留这些核心约束。我还要求 AI 在修改前说明文件范围和测试证据，并用隔离输入做冷启动验证。相比自由发挥，这种约束让我更容易参与和检查过程。

## 凭据与真实模型带来的工程思考

项目默认使用 MockLLM，因此自动验收不需要 API key；但 Web 设置页以及 NJU SE Hub 的 `qwen3.7-max`、`kimi-k2.7-code`、`glm-5.2`、`deepseek-v4-pro` 四模型真实 smoke test，证明凭据边界并不是假设问题。早期实现只有 `fail-closed` 存根，后续 CLI 改为进程环境变量只读优先和操作系统 keyring 持久化；Web 使用独立主密钥与 AES-256-GCM，旧 HMAC 状态迁移后要求重新录入。SpecGate 不读写 `.env`，日志、响应和 trace 也不能回显明文。

这让我意识到，mock-first 不等于可以推迟安全设计。凭据一旦进入配置、数据库或错误处理路径，就必须明确存储、迁移、脱敏和失败关闭规则。PR #22 还暴露了连接测试固定 10 秒造成的假超时：真实模型能正常运行，但前端测试先失败。把测试改为使用已配置的请求超时后，连接测试才与实际运行边界一致。

## 交付失败如何改变我的判断

安全凭据阶段的本地测试、`unit-test` 和 Docker 构建都通过，但 PR #12 合并后的 Pages 仍因 workflow 没有安装项目依赖而失败；PR #13 补充依赖和回归测试后才恢复正常。这让我意识到，“测试通过”不等于“产品可交付”，CI/CD 配置本身也是需要验证的代码。失败、定位、修复和复验的记录，比只展示最终全绿更能体现 Harness Engineering。

Gate/HITL 加固也说明，基础组件存在不代表业务链路正确。即使审批队列实现了 CAS，Runner 或 Web 层仍可能绕过它，因此“真正暂停、批准动作只应用一次、恢复后重新执行最终 Gate”必须由跨层测试证明。正确性不只存在于单个类或函数中，也存在于状态如何跨模块流动。

## 如果重做会改变什么

如果重做一次，我会更早确定 Pages、交互式 Web 服务和 CI 的分工，并提前验证完整交付链。Docker 权限、代理、镜像拉取和 Pages 依赖安装都不应留到最后检查；`REFLECTION.md` 也应该随阶段记录判断变化，而不是最终集中补写。

## 对 Superpowers 方法论的批判

Superpowers 通过 SPEC、PLAN、TDD、验证和日志减少 AI 的随机发挥，也让过程可以追溯。但它的成本是文件、检查点和流程较多，小改动如果机械套用会显得过重；错误计划还可能被 AI 高效执行。因此我认为应按风险选择流程重量：目标明确、可测试的高风险任务适合完整流程，探索性需求则应先澄清和试验。

## 最终总结：我如何理解 Coding Agent Harness

经过这个项目，我把 Coding Agent Harness 理解为连接模型与工程环境的控制层。它把上下文选择、动作协议、工具权限、人工审批、客观反馈、停止条件、恢复和审计组织成可测试的闭环。MockLLM 提供可复现的默认验证，真实模型只作为受同一协议和权限约束的可选决策源。模型负责提出候选动作，Harness 负责限定模型能看到什么、能执行什么、何时必须暂停，以及如何证明结果可信。对我来说，最重要的原则是把关键约束写成可重复验证的代码机制，而不是只写进 prompt。
