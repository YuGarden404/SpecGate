# REFLECTION

> 本文件由学生本人完成，AI 只参与润色和结构整理。最终观点和案例由本人确认。

## Superpowers 哪些技能最有用

这次项目里，我认为最有用的是 `test-driven-development` 和 `verification-before-completion`。Action parser、Guardrail、Gate、Runner 等功能都先用失败测试定义验收条件，再实现最小代码；声称完成前还要重新运行聚焦测试和全量回归。它们共同把“看起来没问题”变成了“有命令输出证明没问题”。

## TDD 在 AI 协作下的作用

我把 TDD 看作控制 AI 输出的手段：红灯证明测试能捕捉缺失行为，绿灯证明本轮需求得到满足，全量回归检查旧模块没有被破坏。Gate 初次实现时，失败摘要没有包含“至少 10 个”的修复提示；根因是摘要只截取前几个 issue，节点数量问题被挤出。这个案例说明，TDD 不只检查代码能否运行，也能检查反馈是否真的服务于下一轮修复。

## Subagent 工作流的有效边界

项目早期我主要使用 Gemini 做冷启动验证：只提供 `SPEC.md` 和 `PLAN.md`，不提供聊天记录，让它检查计划能否被陌生 agent 独立执行。它指出了测试目录隔离和凭据规格精细度不一致的问题，帮助我发现文档依赖了对话中的隐含共识。

`Context Harness Deepening` 阶段后来使用了独立的实现、规格和质量审查 agent；Gate/HITL 之后，因为任务会共同修改 Runner、Web 状态和过程材料，我选择在主线程按 `Inline Execution` 连续推进。这个变化让我觉得 subagent 是否有效取决于任务耦合度：边界清楚的审查和独立任务适合委派，共享状态较多的任务则更需要连续上下文、人工确认和 TDD。

## SPEC / PLAN 质量如何影响实现

项目早期最容易混淆的是两类 SPEC：根目录 `SPEC.md` 描述 SpecGate harness，示例目录中的 `TASK_SPEC.md` 和 `CHECKLIST.md` 才描述一次 HTML 任务。明确这条边界后，agent 没有把项目规约当成运行输入。`PLAN.md` 进一步把工作拆成文件、失败测试、预期结果和验证命令，使实现能按可检查步骤推进，而不是边写边猜。

## 最有效的 prompt / context 策略

我觉得最有效的策略是先明确 MVP 边界：使用 Python CLI、`MockLLM`、静态 HTML 和 Checklist/Gate 闭环，不开放 shell，也不做 Playwright。后续增加 Web 产品壳时仍保留这些核心约束。我还要求 AI 在修改前说明文件范围、作用和测试证据，并只用 `SPEC.md`、`PLAN.md` 做冷启动验证。相比自由发挥，这种 context 控制让我更容易参与和检查过程。

## 凭据与分发要求带来的工程思考

这个项目默认使用 MockLLM，因此自动验收不需要 API key，但 Web 设置页和未来 provider 仍然形成真实的凭据边界。早期实现只有 `fail-closed` 存根，后续 CLI 改为进程环境变量只读优先、操作系统 keyring 持久化；Web 使用独立主密钥和 AES-256-GCM，旧 HMAC 状态迁移后要求重新录入。SpecGate 不再读写 `.env`，日志、响应和 trace 也不能回显明文。

这个变化让我意识到，mock-first 不等于可以推迟安全设计。凭据一旦进入配置、数据库或错误处理路径，就必须明确存储、迁移、脱敏和失败关闭规则，否则以后接入真实模型时只会把已有风险放大。

Docker 和 CI 让我意识到，课程项目不能只是“我本机能跑的脚本”。Docker 验证曾因目录权限和网络连接失败，处理本机环境和代理后才完成 `pull/build/run`。把这些失败写进 `AGENT_LOG.md`，比只保留成功结果更能说明真实交付过程。

## 如果重做会改变什么

如果重做一次，我会更早确定 Pages、交互式 Web 服务和 CI 的分工，并验证完整部署链。Docker 的权限、代理、镜像拉取和 Pages 依赖安装都属于工程交付，不应留到最后检查。`REFLECTION.md` 也应该随阶段记录判断变化，而不是最终集中补写。

## 对 Superpowers 方法论的批判

Superpowers 通过 SPEC、PLAN、TDD、验证和日志减少 AI 的随机发挥，也让最终过程可以追溯。但它的成本是文件、检查点和流程较多，小改动如果机械套用会显得过重；错误计划还可能被 AI 高效执行。因此我认为应按风险选择流程重量：目标明确、可测试的高风险任务适合完整流程，探索性需求则应先澄清和试验。

## 最终交付阶段的补充反思

项目后期我更清楚地意识到，A 类 Coding Agent Harness 的重点不是“模型多聪明”，而是 harness 能否稳定控制模型行为。MockLLM 可以确定性验证协议、权限、Gate/HITL、恢复和配置快照；真实 LLM 不应替代可复现的默认测试路径。

上下文管理也从 Context Manifest 深化为 Select/Compress/Isolate，使输入具备可解释、可限制和可测试的边界。Lab 10 Skill 又把文件、工具、Gate 和汇报规则沉淀成可复用流程。因此项目最终交付的不只是一个 CLI，而是一套可复现、可审查的 agent 工程方法。

## Pages 失败如何改变我对“测试通过”的理解

安全凭据阶段的 753 项本地测试全部通过，PR #12 的 `unit-test` 和 Docker 构建也通过，但合并后的 Pages 仍然失败。原因是 Pages workflow 没有安装项目依赖；PR #13 补充依赖和回归测试后才恢复正常。这让我意识到，“测试通过”不等于“产品可交付”，CI/CD 配置本身也是需要验证的代码。“失败—定位—修复—复验”的记录比单独展示最终全绿更能体现 Harness Engineering。最终 PR #16 合并后，CI #45 和 Pages #27 均通过，形成完整闭环。

## Gate/HITL 加固中我对跨层正确性的认识

基础组件存在，不代表业务链路已经正确接通。即使审批队列实现了 CAS，Runner 或 Web 层仍可能绕过它，因此“真正暂停、批准动作只应用一次、恢复后重新执行最终 Gate”都必须由跨层测试证明。这个阶段让我认识到，正确性不只存在于单个类或函数中，还存在于状态如何跨模块流动；以后检查复杂工作流时，我会优先列出这些跨层不变量，再逐项建立测试证据。

## 最终总结：我如何理解 Coding Agent Harness

经过这个项目，我把 Coding Agent Harness 理解为连接模型与工程环境的控制层。它不只是批量评测或隔离沙箱，而是把上下文选择、动作协议、工具权限、人工审批、客观反馈、停止条件、恢复和审计组织成可测试的闭环。模型负责提出候选动作，Harness 负责限定模型能看到什么、能执行什么、何时必须暂停，以及如何证明结果可信。对我来说，最重要的原则是把关键约束写成可重复验证的代码机制，而不是只写进 prompt。
