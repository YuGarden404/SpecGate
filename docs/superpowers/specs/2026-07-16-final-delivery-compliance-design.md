# SpecGate 最终交付合规修复设计

日期：2026-07-16
状态：已由用户分段确认

## 1. 背景

SpecGate 的 Harness 核心、Mock/Stub 确定性测试、凭据治理、CI、Docker 构建与静态 Pages 已具备，但最终交付材料仍有事实过期和课程条款证据不足的问题。当前代码基线为 PR #20 合并后的 `main@c39d101`，审查起点的完整测试结果为 `Ran 908 tests`、`OK (skipped=27)`。

本阶段只修复材料与过程合规，不扩展产品功能，不发布容器镜像，也不部署公网后端。

## 2. 目标

本阶段需要完成以下目标：

1. 将最终证据同步到 PR #18 至 PR #20 和当前实现事实。
2. 让 `PLAN.md`、`AGENT_LOG.md`、最终证据矩阵和提交检查清单相互一致。
3. 在 `README.md` 中列出所有直接第三方依赖的用途、许可证和官方来源。
4. 在 `SPEC.md` 中如实说明 Open Design 的采用状态或偏离原因。
5. 如实补强冷启动验证证据，区分早期计划审查与本阶段补充实现验证。
6. 核对 PR 描述中的 agent、subagent 和人工干预归属。
7. 用确定性文档契约测试防止相同问题再次出现。

## 3. 非目标

本阶段明确不做以下工作：

- 不修改 `src/specgate/` 下的生产代码。
- 不重新设计或重构 WebUI。
- 不发布 GHCR 或 Docker Hub 镜像。
- 不部署公网交互式 Web 后端。
- 不代写或改写学生反思的观点和案例。
- 不伪造早期冷启动、CI、PR、subagent 或人工决策记录。
- 不把静态 GitHub Pages 描述成交互式 Web 后端。

容器公开分发和公网部署将作为后续两个独立阶段处理。

## 5. 证据模型

事实冲突时使用以下优先级：

1. 当前代码与本阶段最新测试结果。
2. Git commit、merge commit 和 PR。
3. CI、Pages 和对应截图。
4. `AGENT_LOG.md` 中的历史记录。
5. 旧计划、旧说明和阶段性测试数字。

历史测试结果可以保留，但必须标注日期或阶段，不能继续称为当前最终结果。PR #18 至 PR #20、当前基线和本阶段最终测试结果必须进入最终证据矩阵。本阶段新增测试后总数可能超过 908，因此 908 只作为审查起点，不预先写成最终数字。

远端事实必须在真实核对后写入。无法核对的 CI、Pages、PR 描述或截图统一标记为待核对，不得写成已完成。

## 6. 冷启动补充验证

2026-07-08 的 Gemini 3.5 记录保留原样，但重新定性为 SPEC/PLAN 可执行性审查。现有记录没有充分证明发生过完整的 1 至 2 个任务实现尝试，因此不能追溯性宣称其已满足课程全部冷启动要求。

本阶段在材料实施前增加一次补充冷启动验证：

1. 使用与主开发 Agent（Codex）不同类型的全新 Gemini Web 会话。Claude Code 因无官方账号且无法连接 Anthropic 服务而退出，OpenCode 官方 Windows x64 二进制在本机无法加载，Gemini CLI 又因账户没有 Gemini Code Assist 权限而认证失败；用户确认改用可正常访问的 Gemini Web。
2. 不导入本对话历史、memory 或口头解释。
3. 只提供更新后的 `SPEC.md`、`PLAN.md` 和固定任务指令。
4. 让 Gemini Web 尝试完成本阶段的 1 至 2 个材料任务，输出任务理解、补丁草案和阻塞点。由于 Web 版不能直接修改 worktree 或运行测试，该结果只能作为补充冷启动证据，不能宣称为完整的 CLI 实现试跑。
5. 明确要求遇到不确定内容立即暂停提问，不得猜测继续。
6. 保存其问题、输出、误读、与预期差异及执行耗时。
7. 根据结果修订 SPEC/PLAN，并记录修订前后的关键 diff。

该记录必须称为“最终合规阶段的补充冷启动验证”，不能描述为 MVP 实现前已经完成的历史事件。若 Gemini Web 没有暂停、没有产出或任务失败，也应如实记录。

## 7. Open Design 处理

当前项目已经包含交互式 WebUI，但现有 SPEC 没有说明 Open Design。修复时必须根据实际历史选择以下一种真实表述：

- 如果确有 Open Design 使用证据，记录所用设计系统、skill 和对应提交。
- 如果没有使用，则明确记录偏离事实、原因、影响和后续改进边界。

不得通过新增一句声明追溯性声称已经使用过不存在的设计流程。本阶段不借此机会重做 UI。

## 11. 异常与不确定信息处理

- 无法验证的远端信息写为“待核对”。
- 许可证不确定时查询官方元数据，不做推测。
- Gemini Web 没有按预期暂停或产出时记录实际结果。
- 已合并 PR 缺少归属信息时列为待人工更新。
- HEAD、PR 或测试数量变化后，在最终验证阶段重新刷新快照。
- 不为了让文档看起来完整而删除失败历史或平台限制。
