# 终态运行审批保护设计

## 问题

当处于 `needs_approval` 的运行被取消时，运行会正确进入 `cancelled`，恢复入口也会拒绝继续执行，但历史审批仍保留为 `pending`。当前审批接口只校验审批本身的状态，因此运行进入终态后，用户仍能批准或拒绝该审批；Web UI 也仍然启用“通过”“拒绝”和“恢复运行”按钮。

由于恢复路径独立要求运行处于 `needs_approval`，获批动作目前不会被真正应用；但取消后仍可修改审批会污染审计历史，并向用户展示无法形成有效工作流的操作入口。

## 设计

保留现有审批记录和审批队列条目，作为运行取消时仍有待处理审批的历史证据。不在 SQLite 与审批队列文件之间新增跨存储的“取消审批”双写流程。

审批是否可操作改为由所属运行状态派生：

- 审批查询公开所属运行的 `run_status` 工作流字段。
- 批准和拒绝操作在修改审批队列或数据库之前，同时要求 `approval.status == "pending"` 且 `run_status == "needs_approval"`。
- Web UI 仅在审批所属运行仍为 `needs_approval` 时启用批准、拒绝和恢复按钮；正常的待审批运行不受影响。

这样既保留不可变的历史证据，也让终态运行在服务端和界面层同时失败关闭。

## 错误处理

对于终态或其他非待审批运行的审批决定，后端必须在写入审批队列前拒绝请求。API 继续使用现有的安全错误映射，不暴露存储细节。恢复接口保留已有的 `needs_approval` 服务端校验。

## 测试

- 已取消运行的待审批记录不能再被批准。
- 已取消运行的待审批记录不能再被拒绝。
- 被拒绝的决定不会改变审批队列的 revision/status，也不会改变 SQLite 审批行。
- 审批 API 响应包含 `run_status`，但不暴露私有运行配置。
- 前端的批准、拒绝和恢复按钮都要求 `run_status == "needs_approval"`。
- 现有的正常审批和恢复测试继续通过。

## 非目标

- 不新增 `cancelled` 等审批状态。
- 不在取消运行时重写历史待审批记录。
- 不改变现有运行取消状态机或恢复状态机。
